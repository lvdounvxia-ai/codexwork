# 【剧梦PRD】小说导入自动校验生成漫剧剧本

## 一、项目背景

### 1.1 当前现状与解决问题

现状：

| 问题 | 具体表现 | 用户原声 |
| --- | --- | --- |
| 小说/剧本文本导入后结构不统一 | 用户上传的内容可能是小说章节、拍摄分场脚本、纯文本、DOCX 或链接文本，字段顺序和命名不稳定 | “我只想把小说丢进去，系统自动帮我拆成能用的漫剧剧本。” |
| 人工拆集拆场成本高 | 需要人工判断第几集、第几场、场景、时间、人物、正文，长文本处理耗时 | “一篇小说拆成短剧脚本太费时间，先分场就要很久。” |
| AI 修正结果容易改写原文 | 通用模型容易润色、扩写或补剧情，不适合格式校验场景 | “我不要它改剧情，只要它把格式补齐。” |
| 修正后缺少原文对照 | 用户无法快速确认 AI 是否误改、漏改或错拆场次 | “最好左边看修改后，右边能对着原文看。” |
| 格式问题缺少可编辑闭环 | AI 输出后仍需要人工微调，但当前预览如果不可编辑会打断工作流 | “生成完我还要能直接改。” |

核心假设：

如果系统提供“小说/剧本导入 → 自动拆集拆场 → Prompt 格式校验 → 修正后预览可编辑 → 原文对照”的完整流程，用户可以显著降低前期剧本整理成本，并提升导入内容进入后续分镜/视频制作流程的效率。

### 1.2 解决方案

本版本提供一个小说导入自动校验生成漫剧剧本网页 Demo。用户可以通过本地上传、链接导入、直接粘贴三种方式导入文本。系统导入后进入 3 秒“拆解中”演示 loading，随后自动生成分集、分场结构，并在编辑页展示“导入修正后预览区”和“导入原文预览区”。

导入修正后的结果由统一 Prompt 约束：只补齐剧本格式结构，不改写原文剧情内容；内部保留 JSON 给代码校验，用户端展示为可读剧本格式。用户可直接编辑修正后预览区，保存后继续下一步。

系统位置：

```text
小说 / 剧本 / 链接 / 粘贴
        |
        v
导入入口页（三种导入方式）
        |
        v
拆解中 Loading（3s 演示）
        |
        v
Prompt 格式校验与结构补全
        |
        +--> 修正后预览区（可编辑、可保存）
        |
        +--> 原文预览区（只读对照）
        |
        v
下一步：贡献要素 / 分镜视频
```

关键差异：

| 当前工作流 | 新方案 |
| --- | --- |
| 用户手动复制文本并人工拆章/拆场 | 系统自动识别集、场、人物、地点、时间 |
| AI 可能改写原文 | Prompt 强约束“不改写原文，只补格式” |
| 生成结果与原文分离 | 左右双预览，可对照检查 |
| 生成后需要外部编辑器修改 | 修正后预览区可直接编辑、保存 |

### 2.2 用户反馈

用户画像 A：短剧/漫剧编剧

- 特征：已有小说或短剧文本，需要快速整理成分场剧本。
- 痛点：手动拆场耗时，AI 改写导致剧情失真。
- 诉求：不要改剧情，只补格式。

用户画像 B：内容运营/项目制片

- 特征：需要快速评估导入剧本可用性，确认集数、场数、阅读时长。
- 痛点：原文与修正结果无法对照，审核成本高。
- 诉求：左边看修正，右边看原文，一眼发现问题。

用户画像 C：AI 漫剧生产工具用户

- 特征：目标是进入后续贡献要素、分镜视频流程。
- 痛点：导入不稳定会影响后续自动生成链路。
- 诉求：导入后直接可编辑、可保存、可下一步。

| 用户画像 | 核心诉求 | 优先级 |
| --- | --- | --- |
| 编剧 | 原文不改写，只做格式补全 | P0 |
| 内容运营 | 左右对照、可快速检查 | P0 |
| 制片/项目负责人 | 统计集数、场数、阅读时长 | P1 |
| AI 漫剧用户 | 修正后继续下一步 | P0 |

## 四、产品模块

### 4.1 用户操作路径

路径 1：本地上传导入

1. 用户进入剧本内容页。
2. 点击“本地上传”选择 TXT/DOCX 文件。
3. 系统读取文件文本。
4. 进入“拆解中”loading 页，展示 3 秒。
5. 系统生成集/场结构和修正后剧本格式。
6. 进入编辑页，左侧展示可编辑修正后预览，右侧展示原文预览。
7. 用户修改修正后内容。
8. 用户点击保存。
9. 用户点击下一步进入贡献要素。

路径 2：导入链接

1. 用户在“导入链接”区域粘贴公开 TXT 链接。
2. 点击“导入链接”。
3. 系统拉取链接文本。
4. 拉取成功后进入 loading。
5. 拉取失败时使用示例兜底，并提示链接导入失败。
6. 进入编辑页。

路径 3：直接粘贴剧本

1. 用户将小说正文或已有剧本文本粘贴到文本框。
2. 点击该行右侧“下一步”。
3. 系统读取粘贴文本。
4. 若文本为空，Demo 阶段使用示例兜底。
5. 进入 loading。
6. 3 秒后进入编辑页。

路径 4：编辑页修正与保存

1. 用户在左侧选择集、场。
2. 系统展示当前场修正后内容和原文内容。
3. 用户在修正后预览区直接编辑。
4. 系统实时同步当前场 display 内容。
5. 用户切换场次前自动保存当前编辑内容。
6. 用户点击标题栏右侧“保存”，更新保存状态。

异常路径：

| 异常 | 触发条件 | 系统行为 |
| --- | --- | --- |
| 上传非 TXT/MD 文件 | 用户上传 DOCX 等浏览器无法直接解析文件 | Demo 使用内置同名样例；正式版走后端 DOCX 解析 |
| 链接为空 | 用户点击导入链接但未输入 URL | Demo 使用示例文本 |
| 链接拉取失败 | 网络错误、跨域、文件不可访问 | 使用示例兜底并标记来源 |
| 粘贴为空 | 用户未粘贴内容点击下一步 | Demo 使用示例兜底 |
| 内容无法识别场次 | 文本无章节/场次格式 | 按段落自动拆分为场次 |
| 用户编辑后切换场次 | 编辑内容未点击保存 | 系统先同步当前 textarea 内容再切换 |

### 4.2 功能拆解

| 模块 | 责任 | 核心能力 | 依赖 |
| --- | --- | --- | --- |
| 导入入口模块 | 提供三种导入方式 | 本地上传、链接导入、直接粘贴 | 文件读取、fetch、textarea |
| Loading 模块 | 承接导入到编辑的状态过渡 | 拆解中、3 秒自动完成、进度动效 | 定时器、解析任务 |
| 解析拆分模块 | 识别集、场和元信息 | 章节识别、场次识别、段落拆分 | 正则规则、Prompt 结果 |
| Prompt 校验模块 | 生成结构化内部 JSON | 不改写原文、补格式、字段归类 | 模型服务 / 本地模拟 |
| 双预览模块 | 展示修正后和原文 | 左侧可编辑、右侧只读对照 | 编辑器、预览渲染 |
| 保存模块 | 保存用户修正内容 | 标题栏保存、切换前同步 | 前端状态、后端存储 |
| 下一步模块 | 进入后续生产链路 | 进入贡献要素 | 后续模块 |

## 五、产品需求详情

### 5.1 导入入口模块

#### 5.1.1 前置条件

用户已进入剧本编辑页，具备导入剧本内容的权限。

#### 5.1.2 功能描述

导入入口模块提供三种导入方式：本地上传、导入链接、直接粘贴剧本。三种方式同屏展示，采用“横排 1 / 横排 2 / 横排 3”的交互结构。用户无需切换 Tab，即可选择最适合的导入方式。

| 功能 | 交互 | 备注 |
| --- | --- | --- |
| 本地上传 | 点击“本地上传”打开文件选择器；支持拖拽 TXT 文件 | Demo 中 DOCX 使用内置样例兜底 |
| 使用示例导入 | 点击“使用示例导入”直接导入示例剧本 | 用于演示完整流程 |
| 导入链接 | 输入 URL 后点击“导入链接” | 失败时 Demo 使用示例兜底 |
| 直接粘贴 | 在文本框粘贴内容，点击右侧“下一步” | 按粘贴内容进入导入流程 |

#### 5.1.3 验收标准

```gherkin
场景：用户通过本地上传导入 TXT
  前置条件：用户位于导入页，浏览器支持 FileReader
  数据来源：用户选择的本地 TXT/MD 文件
  用户动作：点击「本地上传」并选择文件
  结果：系统读取文件文本并进入「拆解中」loading 页
  并列结果：3 秒后进入编辑页
  交互逻辑：导入期间不展示编辑结果，loading 结束后一次性展示双预览
  边界情况：文件为空时使用示例兜底或提示空文件
```

```gherkin
场景：用户通过链接导入
  前置条件：用户位于导入页
  数据来源：用户输入的公开 TXT 链接
  用户动作：输入链接并点击「导入链接」
  结果：系统尝试拉取链接内容
  并列结果：拉取成功进入 loading，拉取失败使用示例文本兜底
  交互逻辑：按钮点击后触发 fetch；异常捕获后不阻断流程
  边界情况：URL 为空、跨域失败、超时、返回非文本内容
```

```gherkin
场景：用户通过直接粘贴导入
  前置条件：用户位于导入页，粘贴框可输入
  数据来源：textarea 中的用户粘贴文本
  用户动作：粘贴剧本内容并点击「下一步」
  结果：系统读取粘贴文本并进入 loading
  并列结果：3 秒后进入编辑页
  交互逻辑：「下一步」按钮放在第三行右侧，与「使用示例导入」按钮右侧对齐
  边界情况：粘贴为空时 Demo 使用示例兜底；正式版应提示用户补充内容
```

### 5.2 Loading 拆解模块

#### 5.2.1 前置条件

用户已通过任一导入入口触发导入。

#### 5.2.2 功能描述

导入后展示“拆解中”loading 页，作为导入解析的过程反馈。Demo 阶段固定展示 3 秒后自动消失，进入编辑页。

| 功能 | 交互 | 备注 |
| --- | --- | --- |
| 拆解中页 | 显示“拆解中”、动效、进度条 | 默认 3 秒 |
| 状态过渡 | loading → editor | 自动完成 |
| 返回保护 | 用户返回上传页时清除 loading timer | 避免自动跳转 |

#### 5.2.3 验收标准

```gherkin
场景：导入后展示 loading
  前置条件：用户点击任一导入入口
  数据来源：已读取的导入文本
  用户动作：触发导入
  结果：页面进入「拆解中」状态
  并列结果：显示 3 秒进度条动画
  交互逻辑：3 秒后自动进入编辑页
  边界情况：用户在 3 秒内点击返回，系统清除计时器并停留上传页
```

### 5.3 剧本解析与 Prompt 校验模块

#### 5.3.1 前置条件

系统已获得导入文本。

#### 5.3.2 功能描述

系统识别导入文本中的集、场、场景、时间、人物、剧情梗概、正文内容。对已有分场脚本优先按“场次：1-1”识别；对小说章节文本按章节和段落拆分。Prompt 约束只补齐格式，不改写原文剧情。

Prompt 核心规则：

| 规则 | 描述 |
| --- | --- |
| 不改写原文 | 不润色、不扩写、不删减、不重排剧情信息 |
| 仅补格式 | 可补集标题、场次、时间、地点、人物、OS、台词格式 |
| 输出 JSON | 内部保留 JSON 给代码校验 |
| 用户端可读 | 前端将 JSON 中 scene_content 转为剧本格式展示 |

#### 5.3.3 验收标准

```gherkin
场景：系统解析已有分场脚本
  前置条件：导入文本包含「场次：1-1」「场景：」「时间：」「人物：」
  数据来源：导入原文文本
  用户动作：用户完成导入
  结果：系统按场次拆分为多集多场
  并列结果：内部生成 episode、scene_description、persone、scene_content JSON
  交互逻辑：用户端不直接展示 JSON，而展示可读剧本格式
  边界情况：缺失时间/地点/人物时填写「未标注」
```

```gherkin
场景：系统解析小说章节文本
  前置条件：导入文本包含「第一章」「第二章」等章节标识
  数据来源：导入原文文本
  用户动作：用户完成导入
  结果：系统将章节转换为「第一集」「第二集」
  并列结果：每集下按段落拆分为一场或多场
  交互逻辑：左侧侧栏显示集数和场数
  边界情况：章节缺失时将全文作为第一集并按段落拆场
```

### 5.4 编辑页双预览模块

#### 5.4.1 前置条件

导入解析完成，系统进入编辑页。

#### 5.4.2 功能描述

编辑页由左侧侧栏、顶部工具栏、双预览区组成。左侧为“导入修正后预览区”，可编辑；右侧为“导入原文预览区”，只读对照。

| 功能 | 交互 | 备注 |
| --- | --- | --- |
| 左侧侧栏 | 展示集、场列表 | 支持切换场次 |
| 修正后预览 | textarea 可编辑 | 场次号和出场人物高亮 |
| 原文预览 | 只读展示 | 保留原始结构和行号 |
| 保存 | 位于修正后预览区标题栏右侧 | 保存当前编辑内容 |
| 下一步 | 位于顶部工具栏右侧 | 进入贡献要素步骤 |

#### 5.4.3 验收标准

```gherkin
场景：用户编辑修正后预览内容
  前置条件：用户已进入编辑页，当前场存在 display 内容
  数据来源：当前场 scene.display
  用户动作：用户在修正后预览区直接输入或删除文本
  结果：编辑内容实时同步到当前场 scene.display
  并列结果：用户切换场次前自动同步当前输入
  交互逻辑：编辑区为 textarea，底层同步高亮层显示场次号和出场人物颜色
  边界情况：用户输入空内容时允许保存，但需保留当前场记录
```

```gherkin
场景：用户查看原文对照
  前置条件：用户已进入编辑页
  数据来源：当前场 scene.raw
  用户动作：用户查看右侧原文预览区
  结果：系统展示当前场导入原文
  并列结果：原文预览不可编辑
  交互逻辑：右侧保留行号和颜色标识，便于与左侧修正后内容对照
  边界情况：原文为空时展示「暂无内容」
```

### 5.5 保存与下一步模块

#### 5.5.1 前置条件

用户已完成导入并进入编辑页。

#### 5.5.2 功能描述

保存按钮位于修正后预览区标题栏右侧，用于保存当前可编辑内容。顶部工具栏不展示保存按钮，只保留“下一步”作为流程推进按钮。

| 功能 | 交互 | 备注 |
| --- | --- | --- |
| 保存 | 点击修正后预览区右侧保存 | 同步 textarea 内容 |
| 下一步 | 点击顶部右侧下一步 | 进入贡献要素步骤 |
| 替换剧本 | 点击后返回导入页 | 清除 loading timer |
| 导出剧本 | 选择导出 TXT/JSON | Demo 阶段仅展示入口 |

#### 5.5.3 验收标准

```gherkin
场景：保存当前修正内容
  前置条件：用户已编辑修正后预览区
  数据来源：textarea 当前值
  用户动作：点击「保存」
  结果：系统将 textarea 值写回当前场 scene.display
  并列结果：保存状态更新为「已保存 HH:mm:ss」
  交互逻辑：保存按钮位于「导入修正后预览区」标题栏右侧
  边界情况：快速连续点击保存时只保留最后一次内容
```

```gherkin
场景：进入下一步
  前置条件：用户已完成导入并至少存在一场内容
  数据来源：当前剧本结构和用户编辑后的 display 内容
  用户动作：点击顶部「下一步」
  结果：系统进入贡献要素步骤
  并列结果：当前 demo 显示进入下一步状态
  交互逻辑：顶部不显示保存按钮，避免与预览区保存重复
  边界情况：无有效场次时下一步置灰或提示先导入内容
```

## 七、风险预警

### 7.1 技术风险

| 风险 | 描述 | 应对 |
| --- | --- | --- |
| DOCX 解析不稳定 | 浏览器端无法可靠解析复杂 DOCX | 正式版接入后端解析服务 |
| Prompt 误改写 | 模型可能润色或扩写原文 | 强化 Prompt、增加 diff 校验 |
| 拆场不准确 | 小说段落边界不等于场次边界 | 支持用户手动编辑和后续重新拆分 |
| 长文本性能 | 大文档渲染和编辑可能卡顿 | 分场懒加载、虚拟滚动 |
| 高亮编辑器错位 | textarea + 高亮层可能滚动错位 | 同步 scrollTop/scrollLeft，统一字体和行高 |

### 7.2 产品风险

| 风险 | 描述 | 应对 |
| --- | --- | --- |
| 用户误以为已生成最终剧本 | Demo 展示的是格式修正，不是创作改写 | 文案强调“只补结构，不改原文” |
| JSON 与用户展示不一致 | 内部 JSON 和用户可读格式存在双层结构 | 保存时同步 display，并保留 generated 供代码校验 |
| 空内容兜底造成误解 | Demo 阶段空输入使用示例 | 正式版改为明确提示 |

### 7.3 上线检查项

| 检查项 | 标准 |
| --- | --- |
| 导入入口 | 三种入口均可触发 loading |
| loading | 3 秒后自动进入编辑页 |
| 拆集拆场 | 示例文本识别为 8 集 24 场 |
| 修正后预览 | 可编辑、可保存、场次/人物高亮 |
| 原文预览 | 只读、可对照 |
| 下一步 | 顶部按钮可进入后续状态 |
| 埋点 | 导入、编辑、保存、下一步事件完整 |
